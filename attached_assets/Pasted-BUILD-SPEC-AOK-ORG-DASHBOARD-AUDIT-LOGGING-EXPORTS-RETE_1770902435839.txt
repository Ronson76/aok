BUILD SPEC: AOK ORG DASHBOARD — AUDIT LOGGING + EXPORTS + RETENTION

Goal
Implement a tamper-evident audit trail and incident reporting system for AOK so organisations (YMCA, housing associations, care providers, local authorities, universities) can view, filter, and export audit records (PDF + CSV) at any time, and configure record retention.

Non-negotiable rules
1) All audit events are written server-side (never trust device time).
2) Audit records are append-only (no hard edits). If a record must change, create a new “correction” event referencing the original.
3) Every audit event must include: event_id, org_id, user_id, actor_id (who did it), actor_role, created_at_utc, ip_address, device_id (if applicable), app_version, event_type, event_payload (JSON), integrity_hash.
4) Provide immutable event chain hashing per org (hash of event + previous_hash) to make tampering detectable.
5) Encrypt data at rest and in transit. Use least privilege access.

Organisation dashboard requirements

1) Filters
- date range
- user
- activity type
- alert status
- event type
- admin actor

2) Views
- User timeline (activity + alerts + contacts + responses)
- Incident report view (single alert with full timeline)
- Org metrics summary (counts: activities, missed checkins, alerts, avg response time)

3) Exports
- Export Incident Report: PDF + CSV
  Include: user identity, activity details, alert trigger, all notifications (with delivery status), all tracking pings, all contact responses, resolution details, and the full audit event list for that incident.
- Export Audit Log: CSV (date range)
- Export Monthly Summary: CSV + PDF

PDF generation
Generate server-side PDF with a consistent template (header with org name + report ID + generated_at_utc). Include a chronological timeline section.

Retention policy + deletion

Default retention: 6 years (2190 days) from event.created_at_utc.
Allow org admins to set retention_policy_days (min 365, max 3650+ depending on plan).
Implement a scheduled job:
- daily, find audit_events older than retention_policy_days and purge OR archive to cold storage.
- Always keep anonymised aggregate metrics even after purge.
- For user deletion: soft delete user profile but keep audit_events, replacing personal fields in exports with “Deleted User” if required by org policy.

Security + integrity
- Role-based access control (org_admin, auditor_readonly, manager, standard_user).
- Every export logs ADMIN_EXPORTED_REPORT in admin_actions + audit_events.
- Integrity verification endpoint: verify hash chain for a date range; show “pass/fail” on dashboard.

Implementation detail
Use Postgres or equivalent. Use server timestamps (UTC). Provide API endpoints for:
- POST /audit/event
- GET /audit/events?filters…
- GET /reports/incident/:alert_id/pdf
- GET /reports/incident/:alert_id/csv
- GET /reports/audit/csv
- GET /reports/monthly/pdf
- GET /reports/monthly/csv
- POST /org/settings/retention

Acceptance criteria
- All listed event types are recorded.
- Exported incident reports match dashboard timeline 1:1.
- Hash chain verification passes unless records tampered.
- Retention job runs daily and respects org policy.
